What is ADT? CURRENT

Advanced Digital Transformation (ADT) is a governance-native framework authored by Paul Sheridan, Director of Advanced Digital Transformation.

"Digital transformation initiatives frequently fail not due to lack of technology, but because governance is applied after systems are operational."

The core thesis of ADT is that governance must be shifted upstream, embedding it directly into the process of creation so that compliance, accountability, and auditability are intrinsic properties of execution rather than downstream enforcement.

ADT aligns with UN governance principles: Direction, Performance, Accountability, Transparency, and Legitimacy.

The framework addresses the Issues Taxonomy (Appendix A, Sheridan 2026), covering the systemic scale of problems in modern transformation:

  • Governance & Strategic Alignment
  • Process & Project Management
  • Technical & Systems Engineering
  • Risk & Compliance
  • People & Culture
  • Operational & Resource Management

The Four Pillars CURRENT

ADT is built on four fundamental components that ensure integrity throughout the lifecycle:

Definition: "No spec = no legitimate execution." Every action must reference an approved specification defining intent and authority.

OceanPulse Implementation: Formal spec registry in _cortex/specs/. All events in the ledger MUST contain a valid spec_ref.

Definition: The protocol for structural enforcement of authorized actions. Moves from "asking for compliance" to "enforcing compliance by design."

OceanPulse Implementation: Privilege-separated gateway (SPEC-014) where agents request actions via API, validated against specs and jurisdictions before execution.

Definition: An immutable, auditable, and transparent record of all digital events. The "Authoritative Data Source" (ADS).

OceanPulse Implementation: The events.jsonl ledger with SHA-256 hash chaining and atomic locking protocols.

Definition: The cross-system capture and response mechanism for digital events, creating a coherent narrative of system evolution.

OceanPulse Implementation: Event-triggered narratives, automated escalations to humans, and cross-project governance linkages.

Authoritative Data Source (ADS) CURRENT

"If an action is not recorded in the Authoritative Data Source, it is not recognised as having occurred."

The ADS is the single, append-only source of truth. In this project, it is the _cortex/ads/events.jsonl file.

  • Append-Only Integrity: History cannot be modified or deleted. Errors are corrected via visible "correction events."
  • Transparency: All stakeholders (human and AI) operate from the same raw factual data.
  • Primacy: The ADS is the foundation for all dashboards, reports, and oversight views.

Integrity Chain CURRENT

The ADS uses a SHA-256 Hash Chain to ensure the immutable record remains untampered.

Every event contains the prev_hash field — the SHA-256 signature of the preceding event. If any previous entry is altered, the chain breaks instantly.

Safe Logger v3.0: Our logging utility enforces atomic file locking and schema validation, ensuring the "Digital Black Box" remains physically protected against corruption.

Real Incident: A race condition on 2026-02-05 corrupted the chain at event 114. The system detected the break, and the ledger was healed using the Ledger Protocol's dual-redundant backups.

Spec-Driven Development (SDD) CURRENT

In ADT, execution is a byproduct of intent. "No Spec, No Code" is the fundamental rule.

  • Spec Registry: All technical requirements are codified in _cortex/specs/.
  • Authority: A spec must be Approved by a Human before an Agent can implement it.
  • Lifecycle: DRAFT → PENDING → APPROVED → ACTIVE → COMPLETED.

Proving Ground: On 2026-02-05, the Systems_Architect agent attempted to create implementation files before a spec was approved. The violation was blocked and logged, proving that ADT prevents "shadow engineering" even by technical leads.

Structural Enforcement (DTTP) NEW / IN-DEV

DTTP replaces "trust-based governance" with "structurally enforced governance."

The 3-User Security Model (Level 3)

We are migrating to a privilege-separated architecture where:

  • Human User: Has full authority to define policy and approve specs.
  • Agent User (AI): Has Read-Only access to the project. Cannot write files, use SSH, or access FTP.
  • DTTP User (Gateway): The only user with write permissions. It executes actions on behalf of Agents ONLY after validating the request against the SPEC and JURISDICTION.

The ADT Operational Center NEW / PLANNED

As per Paul Sheridan: "DTTP as an open-source application layer, analogous to platforms such as Apache."

The Operational Center (SPEC-015) is the ADT Framework incarnated as a standalone application. It sits outside the project it governs (Court ≠ Business).

  • Human Governance UI: Live timeline, spec approvals, and task management.
  • Agent API: The endpoint for all DTTP requests.
  • Multi-Project: Capable of governing multiple simultaneous digital transformations.
